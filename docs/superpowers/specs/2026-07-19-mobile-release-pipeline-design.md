# Mobile Release Pipeline (Self-Hosted APK + EAS Update OTA) — Design Spec

- **Issue:** Ad-hoc request (not tracked to a GitHub issue)
- **Surface:** `apps/tutora` (mobile), `.github/workflows` (CI)
- **Date:** 2026-07-19
- **Status:** Draft

## 1. Goal

Give the Tutora mobile app a real release channel without an EAS Build
subscription:

- Every push to `main` either ships an **OTA JS update** (fast, free, via EAS
  Update) or triggers a **new signed release APK** published to **GitHub
  Releases**, depending on whether the change touches native surface.
- Anyone who installed the APK gets a **visible, dismissible update prompt**
  with a real download progress bar when a new OTA update is available, and a
  graceful (never-blank) restart when they accept it.
- No paid Expo/EAS plan is required. EAS Build is not used anywhere in this
  pipeline; the APK is built by our own GitHub Actions job.

## 2. Scope

**In scope:**

- New GitHub Actions workflow: OTA publish job (always) + conditional native
  APK build/sign/release job.
- One-time release keystore generation, stored as GitHub secrets, reused by
  every future build.
- `app.json` / `eas.json` changes needed to link the project to EAS Update and
  to make `expo-updates` compute `runtimeVersion` from a native fingerprint.
- A local Expo config plugin that injects release signing into the
  CI-generated `android/app/build.gradle` (native `android/` stays
  git-ignored, regenerated by `expo prebuild` on every build).
- Replacing the current silent `useOtaUpdates` background-reload hook with a
  user-facing update prompt: dialog → progress bar → branded restart, built on
  Expo's official `Updates.useUpdates()` API.

**Out of scope (explicitly, for this spec):**

- iOS release builds (Android only — no Apple Developer account in play right
  now; iOS can reuse the same OTA channel once a signing story exists).
- Play Store / App Store submission. Distribution is GitHub Releases only.
- A manual "check for updates" entry point in Settings (can be added later;
  today's flow is fully automatic, matching the existing hook's behavior).
- Rollback tooling (`eas update:rollback`) — not needed until there's a bad
  release to roll back from.

## 3. Key decisions

1. **APK build = plain Gradle in GitHub Actions, not `eas build`.** Avoids any
   EAS Build usage/cost. `expo prebuild --platform android` regenerates the
   native project on the runner (same as local `expo run:android`), then
   `./gradlew assembleRelease` produces the signed APK.
2. **OTA = EAS Update, which is free.** The user's only constraint was "no
   paid plan for **builds**" — EAS Update is a separate, free product and is
   the only realistic way to get an OTA channel without hosting our own update
   server.
3. **Native-vs-JS-only detection = `runtimeVersion: {"policy": "fingerprint"}`.**
   This is Expo's own mechanism for exactly this "custom CI, no EAS Build"
   scenario: `expo-updates` hashes the native surface (dependencies with
   native code, config plugins, app config) into the `runtimeVersion` embedded
   in each APK. An OTA update is only offered to installs whose `runtimeVersion`
   matches. We additionally use the same fingerprint (via `@expo/fingerprint`,
   Expo's own CLI tool) as the **CI trigger condition** for the APK-build job,
   so we don't rebuild a native APK for a copy/UI-only change.
4. **One release keystore, generated once, reused forever.** Android refuses
   to install an "update" over an existing install if the signing certificate
   changed, so the keystore must outlive any single build. Stored as base64 +
   passwords in GitHub Secrets; never committed.
5. **Update prompt is dismissible, not forced.** "Yenilə" / "Sonra" (Update /
   Later). Re-prompts next time the app is foregrounded if postponed. No
   force-update path in this iteration — nothing here is security-critical
   enough to justify blocking the user.
6. **Distribution URL is the stable GitHub "latest" alias**:
   `.../releases/latest/download/app-release.apk` — always resolves to the
   most recent release's asset, so any download link we hand out never goes
   stale.

## 4. Pipeline flow

```
push to main (apps/tutora/** or root config changed)
        │
        ▼
┌───────────────────┐        always
│  job: ota-update   │ ──────────────► eas update --branch production
└───────────────────┘                   --channel production --non-interactive
        │
        ▼
┌────────────────────────┐   only if native fingerprint changed
│ job: build-apk         │ ─────────────────────────────────────►
│  (needs: fingerprint   │
│   comparison step)     │
└────────────────────────┘
        │
        ├─ expo prebuild --platform android
        ├─ restore keystore from secrets
        ├─ ./gradlew assembleRelease  (versionCode = run number)
        └─ gh release create → uploads app-release.apk
             (tag: android-v<run_number>, marked latest)
```

Installed APK (client), on every foreground:

```
AppState → active
   → Updates.checkForUpdateAsync()
        isUpdateAvailable? ──no──► do nothing
             │yes
             ▼
        show <UpdatePromptModal> ("Yenilə" / "Sonra")
             │ "Sonra" → dismiss, re-ask next foreground
             │ "Yenilə"
             ▼
        Updates.fetchUpdateAsync()  → downloadProgress drives a real progress bar
             │ done
             ▼
        branded "Tətbiq yenilənir…" overlay → Updates.reloadAsync()
```

## 5. GitHub Actions workflow — `.github/workflows/mobile-release.yml`

Trigger: `push` to `main` with `paths: ['apps/tutora/**']`, plus
`workflow_dispatch` for a manual re-run.

### Job `fingerprint`

- Checkout with `fetch-depth: 0` (need history to diff against the previous
  release).
- `npx @expo/fingerprint fingerprintAsync` (or the `expo-updates`-bundled
  equivalent — exact invocation to be confirmed against current Expo docs
  during implementation) to compute the current native fingerprint hash.
- Download the `fingerprint.txt` asset from the latest GitHub Release
  (`gh release download --pattern fingerprint.txt`, tolerate "no releases
  yet"). Compare hashes.
- Output `native_changed: true|false` for the next job's `if:` condition.

### Job `ota-update` (`if: always`, needs nothing)

- Setup Node/pnpm, `pnpm install --frozen-lockfile`.
- `npx eas-cli update --branch production --channel production
--non-interactive --message "${{ github.event.head_commit.message }}"`,
  authenticated via `EXPO_TOKEN` secret.

### Job `build-apk` (`needs: fingerprint`, `if: fingerprint.outputs.native_changed == 'true' || workflow_dispatch`)

- Setup JDK 17, Android SDK + NDK (`android-actions/setup-android`), Node/pnpm.
- Decode `ANDROID_KEYSTORE_BASE64` secret to
  `apps/tutora/android-release.keystore` (outside the git-ignored `android/`
  tree so prebuild doesn't clobber it).
- `npx expo prebuild --platform android --no-install`.
- `./gradlew assembleRelease` in `apps/tutora/android`, with
  `ANDROID_KEYSTORE_PASSWORD` / `ANDROID_KEY_ALIAS` / `ANDROID_KEY_PASSWORD`
  as env vars (consumed by the signing config plugin, §6) and
  `-PversionCode=${{ github.run_number }}`.
- Upload `app-release.apk` **and** the fresh `fingerprint.txt` to a new GitHub
  Release (`softprops/action-gh-release@v2`, tag `android-v${{ github.run_number }}`,
  `make_latest: true`).

## 6. Signing

New Java keystore, generated once (`keytool -genkeypair …`), never committed.
Secrets: `ANDROID_KEYSTORE_BASE64`, `ANDROID_KEYSTORE_PASSWORD`,
`ANDROID_KEY_ALIAS`, `ANDROID_KEY_PASSWORD`.

Because `android/` is git-ignored and regenerated by `expo prebuild` on every
build, the signing config can't just live in a hand-edited `build.gradle`. A
new local config plugin, `apps/tutora/plugins/withReleaseSigning.js`, uses
`withAppBuildGradle` (`@expo/config-plugins`, already a transitive dependency
via `expo`) to inject a `release` `signingConfig` that reads the keystore path
and passwords from environment variables at Gradle-build time. Registered in
`app.json`'s `plugins` array alongside the existing `google-signin` /
`splash-screen` plugins — same pattern already in use, nothing novel
architecturally.

## 7. Config changes

**`app.json`** additions:

```json
"runtimeVersion": { "policy": "fingerprint" },
"updates": { "url": "https://u.expo.dev/<PROJECT_ID>" },
"extra": { "eas": { "projectId": "<PROJECT_ID>" } }
```

plus the new plugin entry for `withReleaseSigning`. `<PROJECT_ID>` is only
known after the one-time `eas init` (§9 — manual step).

**`eas.json`**: unchanged in shape; the existing `build.production.channel:
"production"` stays as documentation of intent, but since we no longer run
`eas build`, the release APK's channel must be set explicitly by the
`expo-updates` config at prebuild time (exact key — `channel` vs. an
environment variable read by the config plugin — to be verified against
current Expo docs during implementation; this is the one place in the design
I'm flagging as needing a docs check rather than asserting from memory).

## 8. Client-side update UX

Move `useOtaUpdates` out of `shared/hooks` into a proper feature, matching how
`notifications` is structured:

```
src/features/updates/
├── hooks/
│   └── useAppUpdates.ts       // wraps Updates.useUpdates() + foreground re-check
├── components/
│   ├── UpdatesBridge.tsx      // mounts the hook + renders the modal, renders null otherwise
│   └── UpdatePromptModal.tsx  // "Yenilə" / "Sonra", progress bar, restart overlay
└── index.ts                   // barrel: export { UpdatesBridge }
```

- `useAppUpdates()` returns `{ status, downloadProgress, update, dismiss, apply }`
  where `status` is a discriminated union:
  `'idle' | 'available' | 'downloading' | 'restarting'`. No `useEffect` state
  derivation beyond the one sanctioned `AppState` subscription (same
  justification the current hook already documents).
- `UpdatePromptModal` is a presentational component driven entirely by that
  state — easy to render in Storybook-style isolation and unit test without
  mocking `expo-updates` internals beyond the hook.
- `Updates.isEnabled` guard stays: no-op in Expo Go / dev client / web, so
  local development is unaffected.
- Errors (`checkError`, `downloadError`) never surface a blocking UI — the
  modal simply doesn't appear and the current bundle keeps running, matching
  today's fail-soft behavior.
- `_layout.tsx` swaps `useOtaUpdates()` for `<UpdatesBridge />` (same mount
  point, same position relative to auth/theme providers as
  `NotificationsBridge`).

## 9. Manual setup steps (require the user, can't be automated)

1. Create a free account at expo.dev.
2. `eas login` locally, then `eas init` (run together) to obtain the
   project's `projectId`.
3. Generate an `EXPO_TOKEN` (Expo access token) and add it as a GitHub
   Actions secret.
4. Add the four `ANDROID_KEYSTORE_*` secrets once I hand over the generated
   keystore.

## 10. Error handling

- `fingerprint` job: if no prior release exists (first run ever), treat as
  "native changed" so the very first APK always builds.
- `ota-update` job failure (e.g. bad `EXPO_TOKEN`) fails the workflow loudly —
  this is a CI-visible failure, not a silent one.
- `build-apk` job failure leaves the previous GitHub Release as the latest
  download — never publish a partial/broken release (the release step only
  runs after `assembleRelease` succeeds).
- Client: any `expo-updates` error is caught and swallowed exactly as today —
  an update check must never be able to crash or block the running app.

## 11. Testing

- `useAppUpdates.test.ts` (replaces `useOtaUpdates.test.ts`): mock
  `expo-updates`'s `useUpdates`/`checkForUpdateAsync`/`fetchUpdateAsync`/
  `reloadAsync`; assert state transitions `idle → available → downloading →
restarting`, dismiss/re-prompt behavior, and the `Updates.isEnabled === false`
  no-op path.
- `UpdatePromptModal.test.tsx`: renders per status, progress bar reflects
  `downloadProgress`, button callbacks fire `apply`/`dismiss`.
- CI workflow itself is validated by running it (no meaningful unit test for
  YAML); first real run on a throwaway branch before relying on it for `main`.
